# Prompt 05/08：信息架构、索引页、导航与错误恢复

## 目标

让用户可以从首页、主导航、页脚、搜索、图谱详情和面包屑到达所有已发布实体；任何公开链接都不得落到 404 或无恢复路径的错误页。

## 范围

只处理路由、导航、列表/索引页、面包屑、404/错误恢复和相关测试。复用现有组件和设计令牌，不重写视觉系统；不扩充数据库内容。

## 现状必须修复

- Header 把 Theories/Scholars/Topics 指向搜索参数，Footer 却指向不存在的 `/theories` 与 `/scholars`。
- 已有 `/[slug]` 详情页，却没有实体索引页。
- 数据故障后的错误页没有可靠站内导航。
- 有些实体页面的面包屑将类别指向搜索参数，结构不稳定。

## 实施要求

1. 创建或恢复下列公开索引路由（以实际一期已发布实体为准）：
   - `/disciplines`、`/fields`、`/theories`、`/scholars`、`/works`、`/topics`、`/concepts`。
   每页必须提供清晰标题、简短说明、实体卡片/列表、空状态及分页或合理上限；不要一次性加载无限内容。
2. 主导航与页脚必须只链接到真实路由。若某一实体类型尚未在一期公开，隐藏该链接或标注为未开放，不能保留死链。
3. `EntityCard`、图谱详情的 `Read Full Article`、搜索结果、面包屑、相关链接必须使用同一套 canonical path 规则；避免一处走 `/search?q=`、另一处走不存在的 index。
4. 详情页的不存在 slug 必须给 404 页面；服务不可用给可恢复错误页；两者均包含首页和相关索引链接。
5. Framework Builder 目前只有“Phase 2”文案。除非本步骤真的实现功能，否则从主导航和页脚移除，或以明确的不可点击“Planned”状态呈现；不要让用户点入后才得知未实现。
6. 补自动路由/链接测试：扫描 Header、Footer、站内列表、面包屑和主要 CTA 的 href；对每条公开路径进行可达性检查。浏览器验收至少覆盖首页、六类索引、一个详情、404、服务不可用页和移动导航。

## 验收标准

- `/theories`、`/scholars` 不再是死链。
- 从 Header/Footer/图谱详情/搜索至少各能走到相同的真实实体详情 URL。
- 公开页面没有“返回图谱”的纯文本假链接。
- 页面在数据为空与服务不可用时语义不同且可恢复。
