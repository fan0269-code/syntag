# Prompt 06/08：完成图谱筛选、节点详情与搜索体验

## 目标

将首页从“只展示 5 个演示节点的画布”完成为可探索真实理论关系的工具：已公开学科、三种图谱模式、节点详情、文章跳转和搜索结果均与实际数据库数据一致。

## 前置条件

Prompt 03、04、05 已通过：数据库已 seed（或至少有一个可用于集成验收的环境）、API 故障语义明确、实体 index/详情路径已存在。若不满足，不要用更多前端 mock 掩盖后端问题。

## 范围

`src/components/graph/`、首页、搜索组件、搜索结果页、图谱 API 投影、与其直接相关的测试。不得更改经过核验的内容事实。

## 实施要求

1. 学科控制仅展示已发布且有可视化数据的学科。若一期只有 Education/Sociology，则 Psychology/Management 不应作为可点击的空选项；若产品需要预告，使用明显 disabled/coming-soon 文案并保证不会发起 404/503 请求。
2. 三个模式必须使用真实数据并有一致定义：
   - genealogy：理论及理论谱系关系；
   - scholars：理论—学者和学者关系；
   - topics：研究主题—理论适切性关系。
   每个模式显示节点数、可解释的空状态和关系标签；禁止用同一份 sampleGraph 假装不同模式。
3. Canvas 节点的指针、键盘和屏幕阅读器备用按钮都必须调用同一个选择逻辑。选中后侧栏应显示实体类型、摘要、核心概念（有则显示）和唯一的详情页链接；关闭后状态恢复。
4. 确保链接生成覆盖 theory、scholar、work、topic、concept；若某类节点没有详情页或 slug，不能显示无效 `Read Full Article`。
5. 搜索覆盖实际公开的 theory、scholar、work、topic、concept、field（是否在 UI 分组展示由产品决定，但数据合同须一致）。至少支持 URL 查询、键盘提交、空查询、无结果、服务不可用和结果跳转。
6. 若搜索结果可聚焦图谱，使用 URL 的 `focus` 参数而非隐藏全局状态；目标节点不在当前学科/模式时，明确告诉用户或切换到可解释的视图，不能静默失败。
7. 不做大规模算法替换。现有 Canvas 布局只需保证节点不完全重叠、缩放/拖拽/点击可用、标签在合理缩放级别可读。

## 浏览器验收

- 教育学与社会学的 genealogy、scholars、topics 都可加载真实节点；未公开学科不会产生错误体验。
- 点击一个 theory、一个 scholar、一个 topic 节点分别出现正确详情并跳转到可达页面。
- 搜索一个已知理论、学者和无结果词；结果跳转、无结果和服务异常都可理解。
- 375px 宽度下，筛选、详情抽屉和搜索浮层没有横向溢出。
