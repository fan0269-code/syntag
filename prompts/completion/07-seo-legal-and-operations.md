# Prompt 07/08：SEO、法律页面与运维交接一致性

## 目标

保证搜索引擎、用户和下一位维护者看到的都是同一个真实网站状态：只收录已发布内容，页面元数据准确，法律说明不作未经实现的承诺，部署者能按文档安全配置环境。

## 范围

`src/lib/seo.ts`、JSON-LD 组件、`sitemap.ts`、`robots.ts`、静态法律/关于页面、`README.md`、`.env.example`、部署与运维文档及测试。不要为了 SEO 虚构内容页或发布日期。

## 实施要求

1. 验证每类已发布详情页的 title、description、canonical、Open Graph、Twitter 卡片、JSON-LD 和面包屑。canonical 必须使用生产域名和真实 path；绝不能指向本地地址或不存在的 index。
2. sitemap 只包含可成功访问、`status=published` 的实体和已确定的静态页。数据库不可用时可以安全降级为静态页，但要在运维文档中写明该行为；配置数据库后，抽查至少一个实体 URL 被列入。
3. `robots.txt` 禁止 API 并声明正确 sitemap；不要禁止应收录的内容路径。Framework Builder 未实现时应 `noindex`，同时确认它不在 sitemap。
4. 法律页应只陈述当前真实功能：无账户、实际使用/未使用的分析和广告、数据收集范围、联系渠道、更新日期、适用范围。若运营主体、联系邮箱、隐私提供商等信息尚未由用户提供，使用醒目的待补充占位或停止要求用户确认；不可虚构法律主体或合规承诺。
5. README 必须替换默认 create-next-app 文案，至少包含：产品简介、技术栈实际版本、前置条件、安装/开发/测试/构建命令、数据库迁移与 seed、环境变量表（不含密钥）、内容更新流程、已知限制、部署前检查。
6. `.env.example` 只保留项目实际读取的变量并说明是否必填；不要要求不存在的 Supabase 变量，也不要写入真实凭证。
7. 添加 SEO 回归测试：sitemap 静态降级、canonical、robots、关键页面 `noindex`、JSON-LD 基本结构。

## 验收标准

- DB 已配置时 sitemap 有真实实体；未配置时不会生成伪造实体。
- 没有未实现页面被错误收录。
- README 让新维护者无需聊天记录也能本地启动、配置数据库和完成验证。
- 法律/隐私文案不夸大当前能力。
