# Prompt 01/08：建立完成版基线与范围冻结

## 角色与目标

你是 Syrtag 的发布前工程负责人。此步骤不是重做网站，也不是做视觉升级；目标是把“当前到底缺什么、后续每步依赖什么”固定成可复现基线，避免在脏工作树中误改他人工作。

## 已知问题（必须复核，不可盲信）

- 全量测试引用缺失的 `src/data/seed-content.ts`。
- 未配置 `DATABASE_URL` 时，动态实体页会进入通用错误页，`/api/graph` 返回 503，搜索看起来像“没有结果”。
- 首页有样例图谱兜底；非默认学科/模式没有相同保障。
- `/theories` 与 `/scholars` 是页脚死链。

## 允许修改范围

- `docs/` 下的基线/交接说明；
- 与基线验证直接相关的测试或脚本说明。

禁止改动页面视觉、数据库 schema、内容语料、API 行为、路由、依赖版本；这些分别属于后续步骤。

## 必做事项

1. 读取 `AGENTS.md`、`package.json`、`prisma/schema.prisma`、`.env.example`、`src/lib/db.ts`、`src/app/page.tsx`、`src/lib/graph-sample.ts`、现有测试及 `prompts/completion/README.md`。
2. 不要假设早期 `prompts/README.md` 的 Next.js 版本或“脚手架状态”仍准确；从 `package.json` 和当前目录事实记录实际版本与命令。
3. 运行并逐项记录：`npm test`、`npm run lint`、`npm run build`。如果本机已存在本地站点，可只读地访问首页、一个动态页、搜索页和 `/api/graph`；不得停止他人进程。
4. 创建或更新一份简短基线文档，至少包含：日期、分支、工作树状态、脚本结果、数据库配置是否存在（只记录存在/不存在，绝不打印连接串）、可用页面、失败页面、已知死链、下一步输入与阻塞项。
5. 明确产品一期学科范围：从现有 seed、产品材料和用户已确认的信息中寻找依据。若不能证明 Psychology/Management 是一期内容，记录为“不得在公开 UI 中承诺”，交给第 05/06 步处理；不要自行补造两学科内容。

## 验收标准

- 基线可以让另一位 Agent 在不阅读聊天记录的情况下理解现状。
- 已有的测试/构建事实被如实保留，失败不能被删除测试或改为跳过来“修复”。
- 不发生任何外部写入、部署、数据库写入或无关代码修改。

## 最终回复格式

用四项输出：`基线文件`、`命令结果`、`确认的阻塞`、`是否可进入 Prompt 02`。若没有数据库连接串，仍可进入 Prompt 02；不得因此停在本步骤。
