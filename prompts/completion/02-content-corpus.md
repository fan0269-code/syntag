# Prompt 02/08：恢复可追溯的内容语料与内容测试

## 目标

恢复 `src/data/seed-content.ts` 及其验证链路，使 Syntag 的首批知识图谱不再只有标题和一句摘要。内容必须面向英文硕博研究者，既能解释理论，也能说明不适用边界、分析转化和来源强度。

## 前置阅读

先完整读取：

- `docs/superpowers/specs/2026-07-12-content-writing-system-design.md`
- `docs/superpowers/plans/2026-07-12-content-writing-system.md`
- `src/data/templates/*.ts`
- `src/lib/content-validation.ts`（如存在）与 `tests/content-validation.test.ts`
- `prisma/schema.prisma` 与现有 `prisma/seed.ts`

如果设计文档与代码的字段名称不同，先以模板和验证器为准，必要时做最小兼容调整并补测试，不要创建第二套 JSON 合同。

## 范围

只处理内容模板、语料、内容验证、与内容 seed 适配直接有关的文件。不要做 UI 美化、导航改版或数据库凭证配置。

## 数据要求

1. 以现有首批 12 个理论为最低语料；保留其 D3/D2/D1 深度分级。辅助图谱节点可以简洁，但必须非空、可解释且有来源元数据。
2. 每个理论至少遵循 `theory-template.ts` 对相应深度的必需区块：定义、起源、核心概念、谱系、适用/不适用、误用风险；D2/D3 还须有分析维度、数据收集、章节建议、理论适切性写法、阅读路径与来源。
3. 任何精确书目信息、DOI、出版年份、人物身份或关系，只有在可靠原始来源（出版社、DOI 注册方、大学/作者主页、原始出版物）可核验时才填写；无法核验时降为 L2/L3 编辑说明或省略精确断言。
4. `L1_verified` 记录必须带至少一个实际来源；不得用虚构 URL、占位 DOI 或“待补充”伪装为核验完成。
5. 明确区分事实、编辑性研究建议和待核验内容。研究建议不能表述为普适的学术结论。
6. 不要为了凑全覆盖而给 Psychology/Management 编造语料；这两个学科是否纳入一期由有证据的范围决定。

## 实施要求

1. 新建/恢复 `src/data/seed-content.ts`，导出唯一的 `seedCorpus` 和 `validateSeedCorpus` 所需的数据合同；不要将大段内容散落在 `prisma/seed.ts`。
2. 让语料能被 `prisma/seed.ts` 消费，并以 slug 连接理论、学者、著作、主题、概念、关系和核验记录。
3. 保持 seed 数据可审阅：按实体/关系分段，使用稳定 slug 与稳定的关系 ID；不要压成不可读的一行 JSON。
4. 扩充 `tests/content-validation.test.ts`：验证首批理论数量、深度分布、必需区块、谱系说明、来源引用、L1 来源约束、所有关系引用均存在。测试不得只验证“字段存在”，还要验证关键字段非空且互相一致。
5. 若现有 `prisma/seed.ts` 只写入 `what_is_it`，将其视为需替换的旧实现；但数据库实际写入留给 Prompt 03 完成。
6. 运行 `npm test`。可运行类型检查/构建作为补充，但本步骤的门槛是内容测试全部通过。

## 禁止事项

- 不删除 `content-validation` 测试、不降低断言、不把失败改成 skip。
- 不复制未经许可的长篇原文；以原创概述与短引用替代。
- 不声明已向数据库插入数据；本步骤不依赖也不要求 `DATABASE_URL`。

## 验收标准

- 缺失模块错误消失，`npm test` 全部通过。
- 12 个主理论与支持节点的引用关系可静态验证。
- 每条 L1 核验都有真实来源，所有内容字段满足模板。
- 代码审阅者可从语料中分辨“已核验事实”和“编辑建议”。
